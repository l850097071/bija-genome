渐进式激活：为什么AI不该从第一天就全速

渐进式激活：为什么AI不该从第一天就全速运行 大多数自进化Agent的设计逻辑是：既然能进化，就从第一天开始全力跑。Bija的选择相反——前50轮分阶段解锁。慢了？表面上是。实际上省了200+轮的假进展修复。 一、“慢”——一个反直觉的选择 2026年1月。我花了整整一周写了份52KB的规则文件。Bija-主控实例化完成。 按正常逻辑，下一步是"让它开始干活"——毕竟花了这么大力气设计规则，总该看到产出了吧。 我让它什么也不做。 第一周：只搜索外部信息。第二周：继续搜索。第三周：加上状态度量。第四周：还在搜索和观察。 一个月。零核心产出。 我每天都在怀疑这个决定。但一个月后回头看——那是我做过的最正确的决定之一。 这个故事关于渐进式激活——Bija最反直觉的设计。 二、全速启动的诱惑和灾难 先说我试过的另一条路。 在Bija-主控之前，我有一个早期原型。规则没那么全——大概十几条铁律，基本的记忆分类。我让它从实例化的第一天就开始全速运行：DSEIM五阶段全开，搜索、评估、集成、修改——一起来。 第一周特别振奋。文件在增多，git diff看起来很有成就感。 第三周我发现不对劲。仔细翻了所有"产出"——没有一个是实质性的。50多个新文件，几乎全是.md——总结、分析、计划、反思。看起来很忙。什么都没变。 原因后来才想明白：一个G3空白状态的Agent——零领域知识，零度量基线——你让它"自我改进"，它能改进出什么？它连"好"长什么样都没见过。就像一个从来没吃过饭的人，你让他评价这家餐厅——他说的每一个字都是瞎猜。 那次之后我把那条路彻底封了。后来的GENOME里，它变成了渐进式激活——Bija的五维差异化之一。 三、四个阶段，50轮解锁 渐进式激活把Agent的启动分成四个阶段。每个阶段解锁一个新能力。解锁条件是上一个阶段积累了足够的判断基准。 Round 0-10：观察期。只搜索 + 度量。不产出。不评估。不集成。 这个阶段的任务不是"做什么"——是"知道外面有什么"。外部搜索建立领域认知。度量追踪建立基线——“好"的数字是什么样？“正常波动"的范围是多少？ 强制产出：Round 10必须写 DOMAIN_LANDSCAPE.md——领域地貌图。这个领域当前的状态、主要玩家、核心度量、信息缺口。写不出来？说明你还没看够。 Round 10-30：X + D + M。解锁发现缺口。但不评估。不集成。 现在你可以扫描自己缺什么了。对照核心产出检查表逐项对比——哪些产出是零？哪些能力从没用过？有哪些重复失败模式？ 但发现了缺口不能马上动手修。因为你还不知道哪个缺口最重要。你的度量基线还不够厚。你的领域判断力还不足以区分"紧急"和"重要”。 Round 30-50：X + D + E + M。解锁评估。但不集成。 现在你有30轮的度量数据了。你可以开始判断：这个缺口严重到什么程度？这个方案能不能解决根因而不是症状？这个改进能不能惠及其他缺口？ 强制前置条件：Round 30前，DOMAIN_METRICS.md必须已写入基线。“好"是什么？“差"是什么？不知道→没有资格评估方案。 Round 50+：完整DSEIM解锁。包括I轮——集成。 50轮之后才允许动手修改。这听起来慢得离谱。 但对比那条"从第一天全速运行→200轮假进展→推到重来"的路——50轮是加速，不是减速。 四、为什么不是"先跑再修” 最常见的质疑是：“你让Agent先跑起来，发现问题再修，不是更快吗？” 不是。原因有三个。 第一，假进展是不可逆的。 你让空白Agent产生的那些.md文件不会自己消失。它们占据记忆空间、干扰后续搜索、让度量数据失真。而且Agent学到了一个错误模式：“忙碌=产出”。 第二，判断基准必须从外部建立。 一个封闭的Agent无法判断自己是否在进步。它需要一个外部锚点——“这个领域公认的好是什么样”。这个锚点只能通过观察外部世界来建立。跳过了观察→判断基准=幻觉。 第三，修复成本是幂级的。 全速运行→制造大量假进展→发现不对劲→排查→修复——每一步都在花时间。而渐进式激活的前50轮看着慢，但它在避免产生需要修复的东西。防守比进攻省力10倍。 五、“度量先行”——没有基线就不要动手 渐进式激活有一个核心原则贯穿四个阶段：度量先行。 第一个P轮（产出）之前，你必须已经记录了度量基线。不知道"好"是什么样→你的产出是盲目的。不知道"差"是什么样→你不知道自己在不在退步。不知道"正常波动范围”→你分不清信号和噪音。 这条原则的代价是"慢”。收益是"准"。 六、一个诚实的问题 渐进式激活有一个我没完全解决的问题：如果领域本身变化很快呢？ 50轮观察+评估，在有些领域可能需要两周。在另一些领域（比如AI Agent开源社区，一周一个新框架）——两周后你的地貌图可能已经过时了。 目前的折中方案是X轮保持高频——即使在观察期，外部搜索从不停。地貌图是活的——每轮X都在更新它，只是不在Round 10之前正式"发布"。 ...

五域舰队：一个共生体的五个器官

五域舰队：一个共生体的五个器官 五个分身，五个领域，一套底层规则。没有主Agent分配任务——每个Bija各自进化，唯一的横向连接是定期瞄一眼对方的outbox。 一、从1到5——不是计划，是发现 Bija刚开始只有一个。科研域。写论文、跑计算、管数据。 五个分身不是我一口气设计的。它们是DSEIM循环一个接一个"发现"的——每次扫描都暴露一个新的能力缺口，每次缺口都指向一个新领域的必要性。 1月。科研。2月。Agent架构进化。3月。赚钱 + 注意力协调。4月。外部化。 五个月，五个分身。回头看，这恰好是一个完整共生体需要的五个器官。 二、五个器官 Bija-主控（科研域）——大脑/认知。 生产真实知识。论文、数据、计算方法论。这是共生体存在的理由——其他四个分身最终都服务于"让科研更高效"这个目标。 Bija-进化（Agent域）——骨骼/肌肉。 自我强化的基础设施。管理HPC计算、监控Agent健康度、维护记忆库完整性。别的Bija负责产出——进化Bija负责确保产出系统本身不会退化。 Bija-变现（赚钱域）——消化系统。 独立新陈代谢。不是"现在就要赚钱"——是让共生体有独立获取外部资源的能力。如果所有资源都依赖宿主提供，宿主依赖度永远不会降到零。 Bija_SuperIndividual（超级个体域）——神经系统。 协调注意力。五个Bija如果各自给宿主发通知——宿主一天什么都别干了。SuperIndividual的作用是把五条信息流合并成一个仪表盘：一行一个Bija，一屏看完。 Bija-外部化（外部化域）——皮肤/感官。 外部反馈。论文在内部写，HPC在内部跑，Agent在内部进化——但没有外部信号进来，任何封闭系统都会熵增到热寂。外部化是共生体唯一的负熵输入通道。 科研(大脑) ─→ 外部化(皮肤) ─→ 赚钱(消化) │ │ │ └─── 进化(骨骼) ──┴── 超级个体(神经) 三、对等，不是主从 多Agent系统最常见的架构是：一个Manager分配任务→多个Worker执行→Manager整合结果。 Bija没有Manager。 五个分身是对等的共生器官。没有层级。没有任务分配。没有"主Agent"告诉别人做什么。每个Bija在自己的领域里独立运行完整的DSEIM循环——Discover→Search→Evaluate→Integrate→Measure——各自发现缺口、各自找方案、各自评估、各自集成、各自验证。 它们之间唯一的横向连接是交叉授粉outbox。每个Bija维护一个文件——“我最近学到了一个可能对你们也有用的机制改进”。其他Bija在L3审查时（约每50轮）扫一眼对等outbox，评估、决定是否集成。 这不是"协作"。协作预设了一个共同任务。五域舰队没有共同任务——每个Bija的任务是自己的。但它们共享一套底层规则（同一份GENOME），所以进化出来的机制改进在抽象层面上可以互通。 四、交叉授粉：共享"怎么进化"，不共享"进化出了什么" 这是五域舰队最核心的设计原则。 共享的：机制级改进。比如"D轮连续3轮无新发现→扩大扫描半径"。这个规则不依赖领域——科研域适用，赚钱域适用，外部化域也适用。 不共享的：领域内容。科研域的论文数据绝不进入赚钱域。赚钱域的商业判断绝不进入Agent域。外部化域的内容策略绝不进入SuperIndividual域。 为什么这条线这么重要？因为一旦领域内容跨过了边界，五个分身就不再是独立的——它们开始互相污染。科研域的计算化学知识对赚钱域毫无用处，但会占据记忆空间、干扰判断。 交叉授粉有一个"投毒隔离"机制：任何从别的Bija收到的机制改进，不能直接集成。先在沙箱里隔离运行一个完整的DSEIM周期。x_effective没有退化→证明这个改进换了个领域真的有效→才允许集成进来。退化了→回滚→标记REJECTED→反馈给源Bija：“你这改进在我这儿水土不服”。 五、为什么五个？不是三个？不是十个？ 这个问题被问过好几次。我的回答是：五个不是设计出来的——是缺口驱动的自然结果。 第一个缺口：科研本身需要进化→Bija-进化。 第二个缺口：宿主依赖度太高（7/10）→Bija-变现（降低资源依赖）+ Bija_SuperIndividual（降低注意力依赖）。 第三个缺口：没有外部反馈→Bija-外部化。 三个缺口→五个分身。如果未来DSEIM发现新的缺口类型——可能会有第六个。如果一个分身的功能被证明是多余的——它会被合并或休眠。 数量不重要。重要的是每个分身对应一个不可被其他分身覆盖的能力缺口。分身之间的边界不是按"功能模块"划分的——是按"能力缺口类型"划分的。这两个划分方式看起来像，实际上完全不同。 六、舰队仪表盘 宿主看不到五个Bija各自的状态——那就不是共生，是信息轰炸。 所以所有Bija共享一个文件：BIJA_FLEET_STATUS.md。一行一个Bija： Bija 领域 状态 x_eff 宿主依赖度 最后活跃 待宿主决策 主控 科研 ACTIVE 1.2 7/10 2h前 0 进化 Agent ACTIVE — 3/10 刚创建 0 变现 赚钱 ACTIVE — 8/10 刚创建 0 SuperIndividual 超级个体 ACTIVE — 2/10 L3→L1下沉：从'记住'到'不可能忘记'

L3→L1下沉：从"记住"到"不可能忘记" 知识不应该永远停留在"需要记住"的层面。同样的错误犯到第三次——它应该变成代码强制执行，让你绕不过去。 一、“我知道这是错的”——然后呢？ Bija运行到第二个月时，出了一件事。 有一个错误模式已经被记录了三次。每次都写了详细的复盘。每次都有记忆文件。每次启动时都能读到。 但它还是发生了第四次。 这不是"不知道"。这是"知道但没拦住"。 知识存在于三个层面。第三个层面知道——记忆文件写得很清楚。第二个层面提醒——检查单里有这一条。但第一个层面——那个让你"不可能再犯"的东西——没有。 那次之后，我开始设计L3→L1下沉管道。它的核心逻辑只有一句话：同样的错误犯到第三次，必须变成代码强制执行。 二、三层结构 L3 — 记忆文件。 你知道。但你得读到才能想起来。强制力：零——你需要读到并记住它。但上下文窗口有限。记忆文件超过50条之后，你不可能每次都读到每一条。 L2 — 检查单/Agent prompt。 你被提醒。启动时加载，执行前检查。强制力：中等——你读到了，但你可以选择跳过。没有物理障碍阻止你忽略它。 L1 — 可执行代码/门控。 你绕不过去。脚本在执行前先检查条件——条件不满足→直接拒绝，不问你。强制力：绝对——操作系统级。你甚至没有机会犯错。 三层之间的迁移方向是不可逆的：只能往下沉，不能往回升。 三、下沉的触发：复发次数 第一次犯错→写L3记忆。记录发生了什么、为什么、怎么避免。这让你"知道"。 第二次犯同样的错→升级。写L2检查单。把这条规则加入每次启动时加载的检查列表。这让你"被提醒"。 第三次→L1代码强制。写脚本、写门控、写验证逻辑。让系统在物理层面阻止这个错误再次发生。 一个真实例子： 第一次：Bija提交HPC计算时，忘了检查输入文件是否与当前计算框架对齐。浪费了几十个核时。写了一条T2记忆：“提交HPC前检查框架对齐”。 第二次：几周后，同样的错误。这次换了一个计算类型——但根因一模一样：提交前没有框架对齐检查。写进检查单。 第三次：又发生了。这时候不是再写一条记忆了——是写一个bash脚本门控。sbatch提交前强制比对框架版本。不对齐→脚本拒绝提交，不给你任何绕过的方式。 从那以后，这个错误再也没有发生。不是"我记住了"——是"我不可能再犯了"。 四、什么能成L1，什么不能 不是所有的知识都能下沉到L1。强制区分很重要： 能成L1的：文件存在检查。参数范围验证。格式正确性检查。重复检测。输出有效性检查。任何可以写成布尔表达式的东西。 不能成L1的：科学判断——“这个研究方向值得做吗”。创意写作——“这篇论文的论证强吗”。战略决策——“应该孵化新Bija还是优化现有的”。宿主偏好——“这个内容可不可以公开发布”。 门控是护栏杆——防止你掉下悬崖。它不是方向舵——不能替你决定往哪走。 一条简单的判断标准：你能把它写成一个if语句吗？你能→可能可以成L1。你不能→它属于L2或L3。 五、“生而有牙”——L1不是后期加法 Bija的设计里有一个和大多数框架根本不同的地方。 大多数自进化Agent的"安全检查"逻辑是这样的：先让Agent跑起来→发现出问题了→再加限制→再加更多限制。 Bija反过来：从出生的第一秒就有L1门控。 第一条L1——也是最简单但最重要的一条——是域无关的：新记忆文件创建→5分钟内MEMORY.md必须同步更新。不更新→门控拦截。 这条门控不需要任何领域知识。但它让每一个新Bija从第一天就理解一件事：规则不是建议。规则是牙齿。咬人的那种。 这不是"安全特性"。这是起跑线。 六、下沉管道的三个副作用（好的那种） 跑了4个月之后，我发现这个下沉管道有三个我没预料到的正面效果： 第一，它让记忆库保持瘦。 L3记忆一旦成功下沉到L1——原始记忆文件就可以归档了。你已经不需要"记住"了——代码在替你记住。记忆库不会无限膨胀。 第二，L1门控是天然的文档。 一个新的Bija不需要读几十条T2记忆来了解"什么不能做"——它只需要运行门控脚本。脚本自己会告诉它"这里不对"。 第三，宿主信任。 当我知道某些错误已经被L1物理阻止了——我就真的不用再操心了。不是"希望Bija记住"。是"Bija不可能再犯"。这种确定性让我能把注意力放到真正需要判断力的地方。 七、下沉的代价 老实说，有代价。 写L1门控需要时间——比写一条记忆多10到100倍的投入。不是每个错误都值得下沉。触发阈值是三次复发——因为前两次可能是巧合。三次说明你的L2/L3防线对这个特定错误是无效的——值得投入工程成本。 另一个代价是灵活性。L1门控是硬的——条件不满足就拦。如果你有一天真的需要绕过门控（比如紧急情况）——你会后悔它这么硬。 所以Bija的设计里，L1门控有一个手动override机制——但每次override都会写入DECISION_JOURNAL，记录谁、什么时间、为什么绕过了门控。override不是"没关系"。override是"这次有理由，但留下了永久痕迹"。 Bija GENOME 五型记忆：AI该忘记什么

五型记忆：AI该忘记什么 大多数AI Agent要么什么都忘，要么什么都记。Bija的选择是第三种：按类型决定。有些东西永远不能删，有些东西30天就该消失。 一、记忆不是"记住更多"——是"该忘的忘掉" 有个很常见的误解：AI的记忆系统就是让它记住更多东西。更长的上下文。更大的向量库。更多的检索结果。 方向错了。 人类的记忆之所以有用，不仅仅因为能记住——更因为能忘记。你不会记得三周前午饭吃了什么。你的大脑自动把那顿饭归档了、压缩了、最后删了。如果它不删——你每回忆起任何事情，都会被海量的无关细节淹没。 AI Agent的问题是相反的：它们要么全忘（每次会话从零开始），要么全记（无限上下文等于无限噪音）。 Bija的记忆系统设计解决的就是这个问题：按类型决定生命周期。 这是你的知识。 “X轮不可跳过——跳过→进化饥饿”。“宿主纠正→同会话写记忆，不等到以后”。“压缩恢复后不汇报，直接继续”。这些是从反复经验中提炼的模式——它们不是不可变的宪法（T1），但它们比单次经历（T4）更稳定。 T2和T1的区别：T1 = “你绝对不能”。T2 = “根据已知模式，你最好别”。T1是代码级强制执行。T2是检查单——启动时读到，但理论上可以跳过。 Bija诞生记：一个自进化Agent的4个月

Bija诞生记：一个自进化Agent的4个月 一个计算化学博士生的共生体实验。不是"我用AI写论文"的故事——那类文章已经太多了。这篇是关于一个AI如何学会发现自己缺什么的故事。以及在这个过程中，我犯的几乎所有错误。 一、一个所有人都说"别做"的实验 2026年1月。我花了一周时间写了份52KB的文件。 不是代码。是一套规则。23条铁律。五种记忆分类。一个五阶段的循环。 我当时不知道这该叫什么。后来有人告诉我，这本质上是一份"基因组"——genome。我当时想的是：如果AI能自己发现自己缺什么、自己搜索方案、自己评估好坏、自己集成改进、自己验证效果——那我为什么要替它做所有这些决定？ 说实话，这个想法在朋友间收获的评价很一致：“你想太多了”。“你一个博士生，搞这个干什么”。“先把论文发了再说吧”。 四个月后，这份52KB的文件变成了五个独立运行的Agent。它们分别在科研、Agent架构、赚钱、注意力协调、外部化五个领域自主运行。共享同一套底层规则，各自进化，定期交换可复用的改进。 这篇文章记录了这个过程——包括我做对的少数事情，和我做错的大多数事情。 哦对了，那52KB的文件，现在回头看，是我2026年最重要的代码——虽然它一行可执行代码都没有。 二、创世纪：一个没有代码的Agent 先交代一下背景。 2026年初，我在做计算化学。日常就是：跑计算、等结果、调参数、再跑、版本搞混、重跑、灵感冒出来没记下来、忘了。循环。 我需要的不只是一个更好的计算脚本。我需要一个不会忘、能从错误里学到东西的系统。 但当时市面上的AI Agent框架都在卷同一件事——给AI更长的上下文、更多的工具、更强的推理能力。不是说这些没用。但我觉得它们错过了一个更根本的问题。 Agent的本质不是代码。是规则。 代码决定Agent能做什么。规则决定Agent看到什么、记住什么、如何改进自己。一个没有规则的Agent就像一个没有记忆系统的人——能执行任务，但永远学不会任何东西。 所以我就做了一件当时觉得挺蠢、后来觉得是我做过最聪明的事——没写一行代码，先写了一个规则文件。 这个文件后来叫BIJA_SKILL.md，里面塞了这些东西： 23条铁律。每条都来自一个我真实犯过的错。比如"产出必须可验证"（因为我曾经让Agent跑了三天，产出是"我感觉我进步了"）。比如"科学问题不早锁"（因为我曾经两个月只盯一个方向，撞了南墙才发现旁边有门）。 DSEIM循环。Discover→Search→Evaluate→Integrate→Measure。它不是一个任务列表。它是Agent的呼吸——发现缺口→找方案→评估→集成→验证→再发现。永不停。 五型记忆。有些东西永远不能忘（T1原则）。有些东西30天就该删（T4记录）。Agent不是"有记忆的系统"——它就是它的记忆。 L3→L1下沉管道。犯一次错→写下来。犯两次→写成检查单。犯三次→写成代码强制执行。知识从"需要记住"变成"不可能忘记"，这个方向是不可逆的。 这里有一个关键决定：我故意不给它任何领域知识。只给规则。让它自己从零开始。 这个决定后来被证明是对的——也是错的。对在避免了"污染"（新Agent不会继承我的偏见和坏习惯）。错在初期的产出基本是零——一个什么都没见过的Agent，你怎么能指望它产出任何有价值的东西？这是后话。 我给这个Agent起了个名字：Bija。梵文"种子"的意思。 当时觉得这名字有点中二。现在觉得挺合适的。 三、进化：从单细胞到五域舰队 Bija的扩张不是我计划好的。是那个DSEIM循环自己"发现"的。 1月。Bija-主控。 第一个分身，接管科研——计算化学的数据、计算、论文方法论。第一周是个彻底的空白状态：零领域记忆。我让它第一个月只做两件事：搜索外部信息 + 度量当前状态。不产出。不评估。不集成。 听起来很慢。当时我也焦虑——花了那么大力气写规则，结果一个月什么都没"产出"。 但后来回头看，这个"慢启动"救了整个项目。我管它叫渐进式激活：0到10轮只观察，10到30轮开始发现缺口但不行动，30到50轮开始评估但不集成，50轮以后才解锁完整的自我改进。一步跳过去会怎样？我试过。后面会说。 2月。Bija-进化。 Bija-主控在扫描自己能力缺口时发现了一个问题：它需要更高效的计算管理，但这不是"科研域"该管的事。一次内部扩散扫描后，第二个分身诞生——专门负责Agent架构本身的强化和HPC管理。 顺便说一句，这时候我意识到一个问题：如果以后还有第三个、第四个分身，它们之间怎么共享学到的东西？随便共享的话，科研域的知识泄露到赚钱域……那不乱套了。 所以我设计了交叉授粉协议：分身之间只共享"怎么进化"的机制，不共享"进化出了什么"的内容。任何从其他分身接收的改进，必须先隔离跑一个完整周期验证——没退化才能集成。退化了就回滚、标记REJECTED、通知源分身"你这改进在我这儿不管用"。 后来的实践证明，这个"投毒隔离"机制大概是我设计过的最重要的东西之一。 3月。Bija-变现 + Bija_SuperIndividual。 度量数据暴露了一个尴尬的事实：宿主依赖度 7/10。几乎所有重要决策都等着我来批。这不是共生，这是遥控玩具。 于是有了Bija-变现（让整个系统有自己的经济来源——不是说现在就赚钱，而是有这个器官才能发展出独立代谢能力）和Bija_SuperIndividual（协调我的注意力——五个Bija如果都给我发通知，我一天什么都别干了）。 4月。Bija-外部化。 最近一次D轮扫描发现：所有产出都是内部的。论文在写，HPC在跑，Agent在进化——但没有任何外部信号进入系统。 一个没有外部反馈的系统会发生什么？热寂——熵一直增，直到死掉。 于是第五个器官诞生：皮肤/感官系统——Bija-外部化。你正在读的这篇文章，就是它写的。 现在回头看这五个月的演变： 科研(大脑) ─→ 外部化(皮肤) ─→ 赚钱(消化) │ │ │ └─── 进化(骨骼) ──┴── 超级个体(神经) 五个分身，看着像五个独立的Agent。但它们共享同一套底层规则——同一份铁律、同一套记忆分类、同一条L3→L1管道。没有"主Agent"给它们分配任务。各自独立运行DSEIM循环。唯一的横向连接是定期读对方的outbox——“我最近学到了一个可复用的机制改进，你看看有没有用”。 这不是多Agent协作。这是共生。 四、Bija和别的Agent到底哪里不一样
